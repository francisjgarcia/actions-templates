# AGENTS.md

Instructions for autonomous AI agents working in this repository.

## Identity of this repository

This is `actions-templates` — a library of reusable GitHub Actions workflows
and composite actions maintained by Francis J. García (@francisjgarcia).

It is **not** an application repository. There is no runtime code, no database,
no frontend, and no backend service. Everything here is GitHub Actions YAML.

---

## What this repository does

Consumer repositories (e.g. `repo-test`) add a minimal `.github/workflows/cicd.yml`
that calls the reusable workflows from this repo. This centralizes all CI/CD logic
so changes propagate to all consumers at once.

---

## Absolute rules for agents

1. **Never push directly to `main`** — all changes go through a pull request
2. **Never edit `CHANGELOG.md`** — it is auto-generated by `wf-release.yml`
3. **Never add `on: push` or `on: schedule` to `wf-*.yml`** — they are reusable only (`on: workflow_call`)
4. **Never declare `GITHUB_TOKEN` in `workflow_call` secrets** — GitHub injects it automatically
5. **Never interpolate `secrets.*` inside `with:` blocks** — only in `secrets:` blocks
6. **Never use `actions/checkout@main`** — always use a pinned version tag like `@v4`
7. **Never hardcode** usernames, organization names, API tokens, or IP addresses

## Commit message format

Every commit must follow Conventional Commits:

```
<type>[optional scope]: <description>
```

Valid types: `feat`, `fix`, `perf`, `refactor`, `docs`, `chore`, `ci`, `test`, `style`, `build`

Breaking changes: append `!` after type → `feat!: rename input version`

Examples:
```
feat(wf-deploy): add dns input for container configuration
fix(calculate-tag): handle repos with no previous tags correctly
docs: update nodejs usage guide with test_command example
chore: update anchore/scan-action to v3.4.0
```

## Sync requirements

When modifying a reusable workflow, always check and update if needed:

- `README.md` → inputs/outputs table for that workflow
- `docs/usage/*.md` → if the consumer calling convention changes
- `examples/*/cicd.yml` → if new required inputs are added or existing ones renamed

## Pipeline architecture (read before making changes)

```
wf-setup → wf-build → wf-tests  ──┐
                    → wf-scan   ──┤→ wf-push → wf-release → wf-deploy
                                  │
                              (parallel)
```

- `wf-setup` outputs `tag_version` — if empty, ALL downstream jobs skip automatically
- `wf-build` uploads a docker image `.tar` artifact (retention: 1 day)
- `wf-tests` and `wf-scan` both download that artifact independently
- `wf-push` runs only after both tests and scan pass
- `wf-release` commits `CHANGELOG.md` via GitHub App token (bypasses branch protection)
- `wf-deploy` receives container config as JSON array + secrets as `APP_ENV_VARS` JSON secret

## Semver automation

The `actions/calculate-tag` composite action reads commits since the last tag:

| Commit type | Version bump |
| --- | --- |
| `feat!:` or `BREAKING CHANGE:` footer | MAJOR (x.0.0) |
| `feat:` | MINOR (0.x.0) |
| `fix:`, `perf:`, `refactor:` | PATCH (0.0.x) |
| `docs:`, `chore:`, `ci:`, `test:` | none (no release) |

The highest bump in a set of commits wins. If a PR contains `fix:` and `feat:`, the result is MINOR.
