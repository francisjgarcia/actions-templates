## Project overview

`actions-templates` is a centralized library of reusable GitHub Actions workflows
and composite actions. Consumer repositories call these templates from their own
`.github/workflows/cicd.yml` instead of duplicating pipeline logic.

This repo does **not** contain application code. It contains only:
- Reusable workflows (`wf-*.yml`)
- Composite actions (`actions/*/action.yml`)
- Documentation and examples

## Critical rules

### Files you must never edit manually
- `CHANGELOG.md` вЂ” auto-generated by `wf-release.yml` on every release
- `examples/` вЂ” these ARE meant to be updated when workflows change

### Files that must stay in sync
When you modify inputs or outputs of any `wf-*.yml`, you must also update:
- `README.md` вЂ” the inputs/outputs table for that workflow
- `docs/usage/*.md` вЂ” if the change affects how consumers call the workflow
- `examples/*/cicd.yml` вЂ” if the calling convention changes

### Reusable workflow rules
- All `wf-*.yml` must have `on: workflow_call:` as the only trigger
- This repo's own `cicd.yml`, `lint.yml`, `pr-validation.yml` call workflows
  using local refs (`./`) not remote refs (`francisjgarcia/actions-templates/...@main`)
- Consumer repositories use remote refs with `@main` or a version tag

### Secrets in reusable workflows
- `GITHUB_TOKEN` must NEVER be declared in `workflow_call` secrets вЂ” it is injected automatically
- App secrets go in a single JSON secret `APP_ENV_VARS` passed to `wf-deploy.yml`
- `vars.*` (non-sensitive) can be interpolated in `with:` inputs
- `secrets.*` can ONLY be used in `secrets:` blocks, never inside `with:` inputs

### GitHub Actions expression rules
- Never interpolate `${{ github.event.*.title }}` or other external values directly in `run:` scripts
- Always pass external values through `env:` blocks and reference as `$VARIABLE` in bash

## Conventional commits

All commit messages and PR titles must follow this format:

```
<type>[optional scope]: <description>
```

| Type | Semver bump | When to use |
| --- | --- | --- |
| `feat!` or `BREAKING CHANGE:` | MAJOR | Breaking change in workflow inputs/outputs |
| `feat` | MINOR | New workflow, action, or input |
| `fix` | PATCH | Bug fix in existing workflow or action |
| `perf` | PATCH | Performance improvement |
| `refactor` | PATCH | Code refactor without behavior change |
| `docs` | none | Documentation only |
| `chore` | none | Maintenance |
| `ci` | none | CI/CD changes |
| `test` | none | Tests |

## What not to do

- Do not suggest adding `development` branch logic вЂ” this repo uses Trunk Based Development
- Do not add `on: push` triggers to `wf-*.yml` files вЂ” they are reusable only
- Do not hardcode usernames, org names, or tokens in workflow files
- Do not use `latest` tags for third-party actions вЂ” use pinned versions (`@v4`, `@v3.6.1`)
- Do not use `secrets: inherit` without explicit justification
- Do not add SSH deploy logic вЂ” the runner deploys directly via docker context
- Do not edit `CHANGELOG.md` directly
